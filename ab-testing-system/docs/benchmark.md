# 性能测试结果

## 测试环境

- **机器**：Mac mini (arm64)
- **JDK**：Java 17
- **用户池**：1000万
- **测试时长**：60秒

## 测试场景

### 场景1：低 QPS（1000 QPS）

| 指标 | 方案一：贪心 | 方案二：自适应 |
|------|------------|---------------|
| P50 延迟 | 22.1 ms | 20.3 ms |
| P99 延迟 | 28.5 ms | 21.2 ms |
| 最大延迟 | 45.2 ms | 23.8 ms |
| MSE | 0.0001 | 0.0012 |
| CPU | 15% | 10% |

**结论**：低 QPS 下两者差异不大，方案一分布更精准。

### 场景2：中 QPS（5000 QPS）

| 指标 | 方案一：贪心 | 方案二：自适应 |
|------|------------|---------------|
| P50 延迟 | 24.3 ms | 20.5 ms |
| P99 延迟 | 35.8 ms | 21.8 ms |
| 最大延迟 | 68.4 ms | 24.1 ms |
| MSE | 0.0002 | 0.0015 |
| CPU | 45% | 20% |

**结论**：方案一开始出现锁竞争，P99 延迟明显上升。

### 场景3：高 QPS（10000 QPS）

| 指标 | 方案一：贪心 | 方案二：自适应 |
|------|------------|---------------|
| P50 延迟 | 28.6 ms | 20.7 ms |
| P99 延迟 | 52.3 ms | 22.1 ms |
| 最大延迟 | 125.6 ms | 25.3 ms |
| MSE | 0.0005 | 0.0018 |
| CPU | 85% | 35% |

**结论**：方案一锁竞争严重，方案二保持稳定。

### 场景4：流量偏置（前10秒全是"广东"用户）

| 指标 | 方案一：贪心 | 方案二：自适应 |
|------|------------|---------------|
| 恢复时间 | 立即 | 3-5秒 |
| MSE 峰值 | 0.0001 | 0.0085 |
| 最终 MSE | 0.0001 | 0.0015 |

**结论**：方案一实时响应，方案二需要3-5秒反馈调节。

## 性能分析

### 方案一瓶颈
1. **锁竞争**：1024个锁，10^5个组合，热点组合竞争激烈
2. **延迟累积**：HBase(20ms) + Redis(2ms) + 锁等待

### 方案二优势
1. **热路径无锁**：仅哈希计算
2. **延迟稳定**：仅 HBase(20ms)

### 方案二劣势
1. **最终一致**：分布偏差需要反馈调节
2. **调节延迟**：3-5秒才能拉回分布

## 推荐配置

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| QPS < 1000 | 方案一 | 分布精准，延迟可接受 |
| QPS 1000-5000 | 方案二 | 性能优势明显，偏差可接受 |
| QPS > 5000 | 方案二 | 方案一锁竞争严重 |
| 需要实时精准 | 方案一 | 无需等待反馈调节 |
